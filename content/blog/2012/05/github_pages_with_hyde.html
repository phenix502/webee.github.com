--- title: 使用hyde+github pages搭建静态站点 description: > hyde+git+[gitbub pages] = 属于你自己的站点，你会有一种清新的感觉。 created: !!timestamp '2012-05-19 17:30:00' tags: - blog - hyde - jekyll - github --- 这篇文章记录了我用hyde和github pages搭建这个博客的过程。里面包括起因，过程，还有一些有用的资料。 ## 起因 无意中发现有github pages这个玩意儿，然后搜索到[BeiYuu.com](http://beiyuu.com)的博客，第一眼就被他博客的简洁风格震撼到了。仔细看了他的文章，感觉发现一片新的天空，感觉特别激动。 现在是在用[is-programmer](http://www.is-programmer.com)的博客，也用wordpress搭建过博客，本来还想着自己什么时候用django再实现一个，但我还是被静态站点这种方式吸引住了，看来现在就是大干一场的时候了。吸引我的几个地方如下： * 内容全静态，部署简单（github pages） * 借助Jekyll和hyde等工具，可以方便的设计复杂结构的站点 * 利用git来管理，方便网站设计和发布 * 内容编辑更自由，直接写html或者用markdown等各种方式生成 总结一下就是：**自由**，**完全掌控**。 ## Hyde还是Jekyll？ 网上大部分的教程都是说的使用Jekyll，我本来也想用Jekyll做的，但是得知Jekyll是ruby写的之后我犹豫了，因为我更偏向于python。在网上搜索一翻后，很快我发现了Hyde。Hyde完全是Jekyll的python版的孪生兄弟。 我自己并没有用过Jekyll，网上有一些针对它们的比较，下面是不由两篇我觉得比较有用的： * [Jekyll-vs-hyde-1](http://www.distractable.net/tech/static-site-generators-jekyll-vs-hyde) * [jekyll-vs-hyde-2](http://philipm.at/2011/0507/) 总结一下， Jekyll的优点： * 容易上手，有很多现成的模板 * github page后台支持 * 有octopress等基于Jekyll的框架使用 Hyde的优点： * content，layout，deploy分离，站点结构更清晰 * 强大的django模板系统(jinja2等) ## 我的选择：Hyde 在仔细的比较之后，我发现Jekyll和hyde在能力上是不相上下的，但hyde有更让我喜欢的理由： * hyde是基于python的 * hyde更像Django * 我喜欢python和Django 选择了hyde就得了解和学习它，下面是Hyde相关的资料： * [Hyde的诞生](http://ringce.com/blog/2009/introducing_hyde) * [hyde](http://ringce.com/hyde)：原始hyde主页 * [lakshmivyas/hyde github][orig_hyde]：原始hyde，已停止开发 * [new Hyde github][new_hyde]：新版hyde，活跃状态 * [Hyde Powered的站点][hyde_powered]：全是基于hyde的网站 * [Hyde文档](http://hyde.github.com/)：正在建设中 ## 开始设计站点 基本的域名，github配置，Jekyll生成站点等请参考BeiYuu的博文[使用Github Pages建独立博客][1]。 ### Hyde安装，基本使用 参考下面： * [hyde github README](https://github.com/hyde/hyde) * [hyde overview](http://hyde.github.com/) 按照这个过程走下来，会得到一个简易的，但真的设计比较全面的站点。 结构如下： {% syntax bash %} - content - media - css - js - images about.html index.html - blog - projects - portfolio - layout base.j2 macros.j2 site.yaml {% endsyntax %} #### 文件说明 site.yaml：整个站点的配置文件 layout/: 站点布局文件夹，里面基本是一些模板文件 content/: 站点内容放在这，包括post，media/(css,js,images)等 通过修改配置和布局就可以设计成自己想要的样子了。 由于站点是全部静态的，所以基本的东西设计好之后，必须利用外部js来实现动态的内容。其实这也是我很喜欢的一种方式：网站利用api或某种方式提供服务，其它用户利用脚本把需要的功能集成起来实现自己的需求。比如：disqus提供评论，gist提供代码显示，dropbox来放图片等 ### 评论 参见[disqus帮助](http://docs.disqus.com/developers/universal/) ### 代码高亮 我想到的基本有三种方式： 1. 截图，利用图片显示代码 2. pygments 3. [github:gist](https://gist.github.com/) 第一种方式，因为不能复制这个缺点，应该很少用到，呵呵。下面说说后两种方式： **pygments**： 首先要安装pygments，这个在安装hyde的时候应该已经装上了。 然后得生成需要的css文件，使用像下面这样的命令： {% syntax bash %} pygmentize -f html -S native -a .highlight > syntax.css {% endsyntax %} 具体的命令可以pygmentize -h查看。或者查看[Hyde README][orig_hyde]页面和[pygments文档](http://pygments.org/docs/)的介绍。 pygments要配合hyde的syntext插件使用，syntext插件提供一个syntax标签，在文档中我们像下面这样使用： {% syntax django %} {{ "{% syntax bash %}" }} pygmentize -f html -S native -a .highlight > syntax.css {{ "{% endsyntax %}" }} {% endsyntax %} syntax标签可以格式化它之间的代码，配合自定义css和生成的syntax.css，就实现语法高亮了。 具体的完全可以参考hyde最开始生成的那个站点的site.yaml设置和css文件设置。 **Gist**: pygments我觉得主要用在一些临时性的代码片断上，如果是一些有一定意义，需要演示的代码，最好还是嵌入gist来显示，这也体现了功能分离的原则，因为毕竟文档不是放代码最好的地方。 ### feed 这个是可以通过模板自动生成的。hyde自动生成的站点中atom.js/atom.xml是很好的例子。 然后利用[feedsky](http://www.feedsky.com/)等站点绑定你的atom.xml来提供feed即可。 ### 其它 archive，tag，分页等，hyde都提供了很好的支持。由于hyde的文档比较少，所以瑞在最好的学习方式就是参考其它人的网站源代码[hyde powered][hyde_powered]。 ## 总结 折腾了几天，终于成形了，很开心。不过真心不喜欢折腾的感觉啊。 由于对网站布局，css等实在不是很懂，很多都是参考其它网站的，这里要特别感谢: * [BeiYuu.com][beiyuu] * [stevelosh.com](http://stevelosh.com) * [jsebold.com](http://jseabold.net) 本站点（[github](https://github.com/webee/webee.github.com)）是参考它们+自己折腾的结果。 [1]: http://beiyuu.com/github-pages/ [beiyuu]: http://beiyuu.com/ [babyishan]: http://babyishan.com [orig_hyde]: https://github.com/lakshmivyas/hyde [new_hyde]: https://github.com/hyde/hyde [hyde_powered]: https://github.com/hyde/hyde/wiki/Hyde-Powered
